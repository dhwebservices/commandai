# Agent Authentication Implementation

**Status**: RFC-001 security gate implemented, ready for security review

This implementation closes the agent-gateway auth gap flagged in ADR-010.
The system is now structurally complete for agent enrollment and mTLS
authentication, though it still requires security review before production use.

## What was implemented

### 1. Database Schema (supabase/migrations/0004_create_agent_auth_tables.sql)

**agent_enrollment_tokens** - Short-lived, single-use tokens for new agent provisioning:
- Generated by tenant admins via API
- Exchanged once for long-lived credential
- Hashed for storage (no plaintext recovery)
- 24-hour max lifetime enforced at DB level

**agent_credentials** - Long-lived agent identity tied to client certificates:
- Stores cert fingerprint (SHA256) as primary lookup key
- Tracks rotation schedule (30-day default)
- Supports revocation with reason/timestamp
- Enforces rotation happens before cert expiry

### 2. Repository Layer (apps/agent-gateway/src/agent-auth.repository.ts)

`AgentAuthRepository` handles all Supabase persistence:
- `generateEnrollmentToken()` - Creates new token, returns plaintext secret once
- `consumeEnrollmentToken()` - Verifies + marks as used (single-use enforcement)
- `issueCredential()` - Creates long-lived credential tied to client cert
- `verifyCredential()` - Looks up agent identity by cert fingerprint
- `revokeCredential()` - Marks credential revoked with reason
- `findCredentialsDueForRotation()` - Query for rotation job

### 3. gRPC Enrollment Endpoint (packages/proto/agent/v1/agent.proto)

New `Enroll` RPC added to AgentChannel service:
```proto
rpc Enroll(EnrollRequest) returns (EnrollResponse);
```

Agent presents enrollment token + client cert → receives agent_id + tenant_id.
From this point forward, agent authenticates via mTLS cert only (no bearer tokens).

### 4. Auth Interceptor (apps/agent-gateway/src/auth-interceptor.ts)

`verifyAgentCertificate()` - The actual security gate:
- Extracts SHA256 fingerprint from client certificate
- Looks up credential in Supabase
- Verifies not revoked/expired
- Returns agent+tenant identity for gRPC call

**CRITICAL**: This function is called on every StreamIntents/StreamStatus connection.
Any bypass would allow arbitrary clients to impersonate agents.

### 5. Tenant Admin API (apps/api-gateway/src/modules/agents/)

New `/agents/enrollment-tokens` endpoint:
- POST creates new enrollment token (admin-only action)
- Returns tokenId + tokenSecret (shown once, never retrievable)
- TODO: Add auth guard to verify user is tenant admin

### 6. Updated Server Handlers (apps/agent-gateway/src/server.ts)

- `Enroll` handler processes token exchange
- `StreamIntents` / `StreamStatus` handlers now extract real client cert and verify
- Proper error handling with gRPC status codes

## Security Properties

✅ **Single-use enrollment**: Tokens can't be reused after exchange
✅ **Time-bounded**: 24-hour max for enrollment tokens, rotation enforced for creds
✅ **No plaintext secrets**: Token secrets hashed at rest
✅ **Revocation support**: Credentials can be marked revoked with audit trail
✅ **Cert-based auth**: Agents authenticate via mTLS, not bearer tokens
✅ **Fingerprint-based lookup**: Stable SHA256 identifier for cert verification

## What's NOT done (follow-up work)

❌ **Cert rotation job**: Needs background worker to issue new certs before expiry
❌ **Actual cert issuance**: Currently assumes agent generates own cert; need CA integration
❌ **Admin auth guard**: Token generation endpoint needs tenant admin verification
❌ **Rate limiting**: Enrollment attempts should be rate-limited per tenant
❌ **Audit logging**: All auth events should publish to audit stream
